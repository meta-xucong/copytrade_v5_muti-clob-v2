# 04 实施路线图

## 1. 实施原则
- 文档先行，编码后置。
- 每一阶段必须有进入条件、交付物、验证方式和退出条件。
- 不允许把“接入层重构、运行器升级、cutover 方案、pUSD 检查”拆散后无序实施。

## 2. 总体阶段划分

### 阶段 0：准备与冻结
目标：
- 锁定当前基线，确保后续每一步有对照。

任务：
- 记录现有测试结果 `53 passed`。
- 确认官方 V2 文档正文、Contracts、pUSD 页面为当前事实依据。
- 确认运行目录不是 Git 仓库根目录，并制定实施期版本化方式。
- 备份当前 `copytrade_config.json`、`accounts.example.json`、运行脚本和关键状态样例。

交付物：
- 基线记录
- 文档版本冻结

退出条件：
- 后续执行者对当前现状、官方日期和风险边界没有歧义。

### 阶段 1：接入层重构
目标：
- 新增统一 V2 交易网关，隔离业务代码与 SDK。

任务：
- 新增 `ct_clob_gateway.py`。
- 将客户端初始化、挂单查询、下单、撤单、盘口读取迁移到网关。
- `copytrade_run.py` 不再直接依赖 V1 SDK。
- `ct_exec.py` 不再直接依赖 V1 SDK 类型。

交付物：
- V2 网关模块
- 基础单元测试

退出条件：
- 主流程和执行层已不再直接导入 `py_clob_client`。

### 阶段 2：费用与市场参数迁移
目标：
- 删除 V1 fee 逻辑，并建立 V2 市场参数缓存。

任务：
- 删除 `/fee-rate` 查询与缓存逻辑。
- 删除 `fee_rate_bps` 透传。
- 新增 `market_info_cache`。
- 接入 `getClobMarketInfo()` 并标准化字段。
- 将 tick size / min order size 的读取迁移到新缓存口径。

交付物：
- 新市场参数缓存逻辑
- 重写后的 fee/market-info 测试

退出条件：
- 代码路径中不再出现手工注入 `fee_rate_bps` 的行为。

### 阶段 3：pUSD 前置与错误处理增强
目标：
- 不自动 wrap，但把资金准备要求显式化。

任务：
- 新增 pUSD 前置检查与错误分类。
- 在执行失败日志中明确提示检查 pUSD 余额与 allowance。
- 将运行手册中的人工准备步骤转成程序提示。

交付物：
- pUSD 预检查逻辑
- 错误提示增强

退出条件：
- 常见余额类错误可被识别并输出明确引导。

### 阶段 4：运行器与脚本升级
目标：
- 让实际启动入口与迁移后的依赖一致。

任务：
- 更新 `bounded_copytrade_runner.py`。
- 更新 `persistent_copytrade_runner.py`。
- 更新 `windows/*.ps1|*.bat`。
- 增加启动前依赖和配置自检。

交付物：
- 可运行的启动器和 Windows 脚本

退出条件：
- 使用正式入口启动时，不会因 V2 依赖缺失或旧配置假设而静默失败。

### 阶段 5：预发布联调与 cutover 演练
目标：
- 在 `clob-v2.polymarket.com` 完成端到端验证。

任务：
- 使用测试钱包完成完整订单生命周期验证。
- 验证 maker/taker、查单、撤单、状态恢复。
- 演练订单簿清空后的本地状态清理与重建流程。

交付物：
- 联调记录
- cutover 演练记录

退出条件：
- 测试矩阵全部通过，且 runbook 已按真实操作修订。

## 3. 里程碑与验收

### M1：文档完成
判定条件：
- `doc/` 下 6 份文档和 `agents.md` 全部齐备。

### M2：V2 网关接管交易接入
判定条件：
- 主流程和执行层不再直接依赖 V1 SDK。

### M3：V1 fee 逻辑彻底移除
判定条件：
- 代码中不再存在 `fee_rate_bps` 的业务传递链。

### M4：运行入口完成升级
判定条件：
- 监督运行器和 Windows 启动脚本可配合 V2 版本运行。

### M5：预发布验证通过
判定条件：
- V2 环境下完成端到端下单、撤单、状态恢复演练。

### M6：cutover 就绪
判定条件：
- 有正式 runbook，且演练通过。

## 4. 任务分解建议

### 4.1 开发顺序
1. 网关模块
2. `copytrade_run.py` 迁移
3. `ct_exec.py` 迁移
4. `ct_state.py` 状态扩展
5. 测试重构
6. 运行器/脚本更新
7. 联调与 runbook 修订

### 4.2 禁止顺序
- 不允许先删 V1 fee 逻辑再不补网关。
- 不允许先做 cutover runbook 再不做状态清理机制。
- 不允许只改主程序不改运行器和 Windows 脚本。

## 5. 风险与缓释

### 高风险
- V2 SDK 接口细节与当前假设不一致
  缓释：先安装 `py_clob_client_v2`，以小样例验证接口后再改主程序。

- cutover 后挂单清空导致状态错配
  缓释：实施专门的“远端刷新 + 本地影子挂单清理”步骤。

### 中风险
- pUSD 准备不充分
  缓释：上线前必须完成人工资金清单核对，并在程序内增加错误提示。

- 当前目录不是 Git 仓库根目录
  缓释：实施前建立可追溯版本化方式，至少保证每一阶段有可恢复快照。

## 6. 退出标准
迁移不能以“代码能跑”为结束，而应满足以下标准：
- 测试基线保持可复跑。
- 新增 V2 测试通过。
- V2 环境端到端联调通过。
- cutover 演练通过。
- 启动器和运维脚本同步更新。
- 文档与代码一致。

## 7. 路线图结论
本项目按六个阶段推进即可控制风险。任何跳过阶段 1、2、4、5 的做法都会让最终 cutover 暴露出系统性问题，因此不允许压缩为“一次性提交完成”。
