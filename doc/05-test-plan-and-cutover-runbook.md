# 05 测试计划与 Cutover Runbook

## 1. 测试目标
- 确认迁移后程序仍保持原有跟单目标。
- 验证 V2 交易接入层、市场参数获取、错误处理和状态恢复符合设计。
- 在正式 cutover 前完成真实环境演练。

## 2. 测试矩阵

### 2.1 基线回归
目的：
- 确认迁移前后的已有能力没有意外退化。

要求：
- 迁移前记录一次完整测试结果：当前为 `53 passed`。
- 迁移后第一轮回归必须覆盖全部现有测试。

通过标准：
- 现有测试应保持可运行。
- 若因 V2 设计导致测试需要重写，必须在变更说明中记录替换关系。

### 2.2 单元测试
新增测试覆盖以下主题：
- `tests/test_clob_gateway_v2.py`
  验证客户端初始化、查单、下单、撤单、盘口读取的适配行为。
- `tests/test_market_info_cache.py`
  验证 `getClobMarketInfo()` 标准化和缓存行为。
- `tests/test_pusd_preflight.py`
  验证 pUSD 前置校验和余额类错误提示。
- `tests/test_cutover_rebuild.py`
  验证 cutover 后本地状态刷新与挂单影子重建逻辑。

### 2.3 集成测试
覆盖以下路径：
- 账号初始化
- API creds 派生或绑定
- 查询持仓
- 查询挂单
- 下 maker 单
- 下 taker 单
- 撤单
- 重启后状态恢复
- 市场关闭 / 不可交易状态处理

### 2.4 预发布联调
环境：
- `https://clob-v2.polymarket.com`

要求：
- 至少使用一个测试钱包。
- 至少验证官方迁移页中列出的 V2 测试市场之一。
- 验证完整订单生命周期。

### 2.5 运行验收
验证以下业务行为未偏移：
- 目标地址动作识别
- 多目标地址整合
- 多账户轮转
- 风控限额
- 退出逻辑
- 状态持久化
- 启动器监督恢复

## 3. 执行顺序
1. 先跑单元测试
2. 再跑本地集成测试
3. 再做预发布联调
4. 最后做 cutover 演练

禁止倒序执行。

## 4. Cutover 前准备清单

### 4.1 文档与配置
- 核验官方迁移页正文中的 cutover 时间。
- 核验 Contracts 页面中的 V2 地址是否有更新。
- 核验 pUSD 页面与 onramp 说明是否有更新。
- 确认 `copytrade_config.json` 已切换到 V2 所需配置。

### 4.2 环境与依赖
- 安装并验证 `py_clob_client_v2`。
- 验证运行器和 Windows 脚本在目标机器上可用。
- 验证日志目录与状态目录可写。

### 4.3 资金准备
- 人工确认交易账户已有可用 pUSD。
- 若使用代理/资金地址结构，确认 funder 侧资产已准备完毕。
- 如需人工执行 `wrap()`，必须在 cutover 前完成并留记录。

### 4.4 联调结果
- 确认预发布联调通过。
- 确认最近一次回归结果已记录。

## 5. Cutover 当天 Runbook

### 5.1 T-60 分钟
- 停止自动启动新实例。
- 保持现有进程只读观察，禁止再做新配置变更。
- 备份当前配置文件和最新状态文件。

### 5.2 T-30 分钟
- 停止正式跟单进程。
- 记录当前远端持仓、远端挂单、本地状态文件快照。
- 记录运行器会话文件和日志路径。

### 5.3 停机窗口内
- 不重启旧版程序。
- 持续关注官方状态页、Discord 或公告。
- 不尝试用 V1 SDK 恢复交易。

### 5.4 官方恢复后
- 先做一次只读检查：
  API 可达、盘口可读、挂单列表可读、持仓可读。
- 强制刷新远端 open orders。
- 由于官方会清空挂单，应将本地影子挂单视图视为失效。

### 5.5 本地状态处理
恢复前必须执行以下逻辑：
- 清理本地 `open_orders`、`open_orders_all`、`managed_order_ids` 的历史影子状态。
- 保留业务状态、风控状态、目标跟随状态。
- 记录一次 `cutover_rebuild_pending=true`，直到首次成功完成远端刷新和新挂单同步。

### 5.6 恢复运行
- 先以小规模或低风险配置启动。
- 观察首次远端刷新、首次下单、首次撤单、首次状态保存。
- 确认无连续 `balance / allowance / invalid order` 错误后，再恢复正常运行节奏。

## 6. Cutover 后检查项
- 远端挂单视图与本地状态一致。
- 首次新挂单可以成功成交或撤销。
- 持仓同步无异常。
- 日志中没有持续出现 V1 遗留字段相关错误。
- 多账户轮转正常。
- 运行器监督重启正常。

## 7. 异常处置

### 7.1 无法下单
优先排查：
- `py_clob_client_v2` 是否正确安装
- API creds 是否可用
- pUSD 是否准备完成
- `builderCode` 是否误传

### 7.2 持续余额不足
优先判定为：
- pUSD 未准备完成
- allowance 不足
- funder 地址与资金所在地址不一致

处置：
- 暂停自动下单
- 先人工核对资金准备

### 7.3 本地状态与远端挂单不一致
处置：
- 以远端为准
- 清空本地影子挂单状态
- 重新同步一次，不做人工拼接修复

### 7.4 需要回滚
说明：
- cutover 后不能回滚到 V1 继续交易。
- 所谓回滚，仅指回退到“较早的 V2 候选版本”或暂停系统运行。

## 8. 最终验收标准
- 现有测试可复跑。
- 新增 V2 测试通过。
- V2 联调通过。
- cutover 演练通过。
- 运行器和 Windows 脚本已同步升级。
- 文档与实际行为一致。

## 9. Runbook 结论
cutover 成功的关键不只是“代码兼容 V2”，而是：
- 资金准备完成
- 旧挂单影子状态正确失效
- 恢复顺序可执行
- 出现异常时能快速判断是接入问题、资金问题还是状态问题

这四项必须在正式切换前完成演练。
